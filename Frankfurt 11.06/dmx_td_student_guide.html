// 01 — History & origins
Why DMX exists
Not history for history's sake — understanding this explains why the protocol is shaped the way it is, which explains why TD works the way it does.
// lighting control history — from ancient theater to DMX512
KEY INSIGHT
A cue is just a decision
In ancient Greece, someone decided when to pull the rope. That's a cue. The concept hasn't changed — only the tools have.
THE PROBLEM
Proprietary chaos
By the 1980s every manufacturer had a different protocol. A console from one country couldn't talk to fixtures from another.
THE SOLUTION
DMX512: one standard
512 channels, values 0–255, one-way broadcast. Standardized in 1986. Still the backbone of professional lighting 40 years later.
// 02 — DMX Protocol
512 numbers. That's it.
One universe = 512 channels. Each channel = a number from 0 to 255. Refreshed up to 44 times per second.
// universe layout — fixture addressing and channel footprints
ADDRESSING
Start address + footprint
Every fixture has a start address. It listens to that channel and the next N channels it needs. Two fixtures at the same address = both respond identically.
MATH
Channel calculation
Pixel tube, 8px RGB: 8 × 3 = 24 channels. Max per universe: 512 ÷ 24 = 21 tubes. You can't split a fixture across a universe boundary.
TRANSPORT
Art-Net over IP
Same 512-channel frame, wrapped in a UDP packet. Your machine and fixtures need to be on the same subnet (2.x.x.x or 10.x.x.x).
Exercise 01 Address calculation 10 min

Rig: 12 × RGBW PARs  +  6 × pixel tubes (12px RGB)  +  4 × moving heads (16ch)

  1. Calculate total channels for each fixture type
  2. Assign addresses sequentially — where does universe 1 end?
  3. How many more pixel tubes could you add before needing universe 2?
Answer

12 RGBW = 48ch  ·  6 tubes (12px) = 216ch  ·  4 MH = 64ch  →  Total: 328ch (fits in one universe). You could add 5 more pixel tubes (180ch) to reach 508ch — a 6th would overflow to universe 2.

// 03 — Color theory
Why LEDs use RGB
A diode emits light — it can't absorb it. So RGB (additive) is the only model that makes physical sense for a light source.
// additive color mixing + RGBW upgrade + RGB→RGBW conversion formula
PHYSICS
LEDs can only add
A diode emits light — it can't absorb wavelengths like paint or ink. CMY (subtractive) is physically impossible for an LED fixture.
HISTORY
Blue LED — Nobel Prize 1994
Without blue, you can't make white. Before 1994, "white" stage light meant incandescent or halogen. The blue LED made RGB multi-chip possible.
IN PRACTICE
RGBW → cleaner white
Adding a dedicated white phosphor diode gives a warmer, brighter, more efficient white than mixing R+G+B. Extract it in GLSL: W = min(R,G,B).
// 04 — TD Architecture
GPU vs CPU — why it matters
TD runs on a single CPU thread. For pixel-scale operations, that's a bottleneck. The GPU runs thousands of simple operations in parallel. Know which one you're using.

Exercise 02 Basic DMX output in TD 25 min

PART A — CHOP-BASED (understand the mechanism)

  1. Create a Constant CHOP — 3 channels named r, g, b. Values: 0.5, 0.2, 0.8
  2. Add a Math CHOP — multiply range 0–1 → 0–255
  3. Add an DMX Out CHOP CHOP — set IP to your visualizer, Universe 0
  4. Open visualizer — fixture should show your color. Now swap in an LFO CHOP and watch it animate.

PART B — TOP/GPU (scale it)

  1. Create a Constant TOP — resolution 8×1 (8 fixtures)
  2. Add a GLSL TOP with a chase shader, feed absTime.seconds to uTime
  3. Add a DMX Out CHOP CHOP in "Convert from TOP" mode — 8 fixtures, GPU-driven

uniform float uTime;
uniform vec4 uColor;
uniform float uPeriod;

out vec4 fragColor;
void main() {
  float brightness = step(mod(vUV.s * 1/uPeriod + uTime * 2.0, 1.0), 0.5);
  fragColor = vec4(uColor*brightness);
}

Exercise 03 Audio-reactive chase 20 min
  1. Create Audio Device In CHOP or Audio Device In CHOP → Audio Analysis from Palette (8 bands, smoothing 0.3)
  2. Select CHOP → grab channel 0 (low freq / kick)
  3. Lag CHOP — attack 0.01, release 0.3. This is your kick detector.

// 07 — Production & show day
Studio works. Show is different.
Network, redundancy, working with an LD — what you need to know before the first load-in.
// show network architecture — dedicated VLAN, managed switch, redundancy
NETWORK
Dedicated + static
Lighting network is always separate. Managed switch only. Static IPs for everything — DHCP on reboot = disaster mid-show.
REDUNDANCY
Always have a backup
Two TD machines, one standby. Always have a simple backup scene that can be triggered instantly. Never go to air without it.
WITH AN LD
Be explicit about ownership
LDs think in cues and groups. Be clear: what does TD control vs the console. Shared universes cause conflicts. Document it before load-in.
