Audience — TD users, no lighting background

Opening frame

"Today is not a TD tutorial. You already know TD. 01

Where lighting control comes from

20 min

Flashlight exercise

Ask everyone to get their phone out, flashlight ready. Split the room into rows. Signal: left hand up → left row on, right hand up → right row on, both down → all off. Run it a few times. Build a chase pattern.

"I just programmed a chase effect. Using unpaid labor. Like Apple."
Then: "Everything we do today is this. Turning lights on and off with different colors. That's it. The software is just a way to do this faster and with more precision."

From Greece to the 1980s

  • Deus ex machina: rope + lever + actor lowered from above = first recorded staging automation. Someone had to decide when to pull the rope. That's a cue. Cues still exist in every console today.
  • Medieval: torches → chandeliers → gas lamps → electricity
  • Key moment: multiple bulbs at different stage positions → you can accent areas. Birth of lighting design.
  • First programmable desks looked like power station control panels
  • By the 1980s: explosion of proprietary protocols — consoles and fixtures from different manufacturers couldn't talk to each other
// diagram — lighting control history timeline

The dimmer block

It's the interface between the control signal and mains power going to the fixture. Think of it as: your brain (TD) → signal → hand (dimmer block) → controls power → phone flashlight (fixture) = output. A dimmer can also do gradual intensity — 0% to 100% — not just on/off.

Why DMX512

In 1986, USITT standardized DMX512: Digital Multiplex, 512 channels per universe. A one-way serial protocol — the console broadcasts, fixtures listen. Every channel is a number 0–255. This is still the backbone of professional lighting 40 years later.

"DMX is just a list of 512 numbers, refreshed up to 44 times per second. That's it. Every lighting effect you've ever seen at a concert, in a theater, at Eurovision — is ultimately a list of numbers changing over time."

02

DMX protocol deep dive

25 min

The DMX frame

  • 512 channels = one universe; each channel: integer 0–255
  • Refresh rate: up to 44Hz (some controllers go higher)
  • Unidirectional: console talks, fixtures listen — no acknowledgment, no error correction
  • Physical: RS-485 balanced signal, 3-pin or 5-pin XLR cable
  • Today: mostly transmitted over IP via Art-Net or sACN
// diagram — DMX universe: 512 channels as a data strip

Fixture addressing

Every fixture has a start address — the first channel it listens to. It then consumes as many channels as its footprint requires. If you set two fixtures to the same address, they both respond identically — useful for duplicating effects, dangerous if unintentional.

// diagram — fixture addressing in a universe

Channel math

Fixture typeChannelsNote
RGB PAR3R, G, B
RGBW PAR4R, G, B, W
RGBWUV PAR6R, G, B, W, Amber, UV
Pixel tube (8px RGB)248 × 3 — max 21 per universe
Moving head (wash)16–25Pan, tilt, color, gobo, zoom...
Moving head (spot)20–30++ prism, iris, animation wheel
Universe boundary caveat

A fixture's channels must all live within one universe. You can't split a fixture across a universe boundary — if 8 channels remain in universe 0 but your next tube needs 24, it starts in universe 1. This trips everyone up eventually.

Art-Net vs sACN

  • Art-Net: most common, simpler to configure, works over standard ethernet/wifi. Uses broadcast or unicast on 2.x.x.x / 10.x.x.x subnets.
  • sACN (E1.31): newer, multicast-native, slightly more efficient for large systems.
  • In TD: Art-Net Out CHOP, sACN Out CHOP — same from a patching perspective
  • Network config: your machine and fixtures must be on the same subnet. For Art-Net, set IP to 2.x.x.x range.
Network gotcha

If you're using sACN (multicast) and fixtures are mysteriously disappearing, check IGMP snooping on your managed switch. On large shows with Luminex GigaCore or similar, Araneo can monitor this in real time.

Exercise 01 Address calculation 10 min

You have a rig: 12 × RGBW PARs, 6 × pixel tubes (12px RGB), 4 × moving heads (16ch).

  1. Calculate the total channel count for each fixture type
  2. Assign addresses sequentially — where does universe 1 end and universe 2 begin?
  3. Identify which fixture would cross the universe boundary if you added 3 more pixel tubes
Answer

12 × RGBW = 48ch  |  6 × pixel tube (12px RGB) = 6 × 36 = 216ch  |  4 × moving heads = 64ch  →  Total = 328ch (fits in one universe). Adding 3 more pixel tubes: 328 + 108 = 436ch — still fits but getting close. One more tube = 472ch, two more = 508ch, three more = 544ch → universe 2 needed after 512ch, the last pixel tube would be split.

03

Color theory for lighting

20 min

Additive vs subtractive

CMY (subtractive): used in print and paint. Start with white, subtract colors by absorbing wavelengths. Can't work for LEDs — a diode emits light, it can't absorb it.

RGB (additive): used in screens and LEDs. Start with darkness, add light. The only model that makes physical sense for a light source.

// diagram — RGB additive color mixing

History of the blue LED

  • Green LED: invented first (gallium phosphide, relatively simple)
  • Red LED: soon after
  • Blue LED: not until 1994 — Shuji Nakamura, Nobel Prize in Physics 2014
  • Without blue, you can't make white. Before blue LEDs, "white" in stage lighting meant incandescent or halogen bulbs.
  • Multi-chip LED (single die, multiple colors): only became possible after blue was invented

RGBW — why add a dedicated white channel

  • RGB mixing to white produces a slightly cold or dirty white — the three primaries don't reconstruct a broad-spectrum source
  • Dedicated white phosphor diode: cleaner, warmer white; more power-efficient than R+G+B all at full
  • Some fixtures: RGBWUV (+ Amber, UV/ultraviolet) for even wider color gamut and special effects

RGB → RGBW conversion in TD

In pixel pipelines you often have RGB texture data but RGBW fixtures. Extract a white component and redistribute:

GLSL — RGB to RGBW
vec3 rgb = texture(sTD2DInputs[0], vUV.st).rgb;
float w    = min(rgb.r, min(rgb.g, rgb.b));
vec3 rgbOut = rgb - vec3(w);
// Output: R, G, B channels reduced + W channel added
fragColor = vec4(rgbOut, w);
Performance tip

Do this in a GLSL TOP, not in Python. A Python loop over pixel values will tank your FPS above 100 fixtures. The GLSL version runs in parallel across all pixels on the GPU.

Two approaches to DMX output

// diagram — CHOP-based vs TOP-based pipeline

GPU vs CPU — why it matters in TD

CPU analogy: a building with 16 specialist workshops. Each can make anything — custom PCB, ornate wardrobe, whatever. Incredibly capable but slow and expensive per task. TD runs primarily on a single CPU thread — so effectively you have one workshop, not 16.

GPU analogy: a warehouse with 10,000 simple robots. Each can do exactly one thing: take this number, multiply it by that number, put the result here. Can't do anything complex — but they all do it simultaneously. That's what makes textures fast.

The transfer problem: moving data between CPU and GPU has a cost. Every time you read pixel values in Python (TOP.numpyArray()), you're forcing a GPU→CPU memory transfer. The DMX POP was significant because it avoids this entirely.

Optimization rules of thumb

  • Touches every pixel every frame → GLSL shader
  • Runs once at startup or on config change → Python is fine
  • Per-fixture logic (not per-pixel) → CHOP is fine
  • Never use numpy in a perform-mode callback without profiling first
Exercise 02 Basic DMX output in TD 25 min

Two parts. Part A: understand the mechanism via CHOPs. Part B: scale it with a texture.

PART A — CHOP-BASED OUTPUT

  1. Create a Constant CHOP — 3 channels named r, g, b. Set values: 0.5, 0.2, 0.8
  2. Add a Math CHOP — set range 0–1 → 0–255 (multiply by 255). This is your raw DMX value.
  3. Add an Art-Net Out CHOP — set IP to your visualizer (or 255.255.255.255 for broadcast), Universe 0
  4. Open your visualizer — you should see the fixture color you set
  5. Add an LFO CHOP and merge it — you now have animation
  6. Try a Pattern CHOP (ramp) across multiple channels — you now have a chase

PART B — TEXTURE-BASED (GPU)

  1. Create a Constant TOP — set resolution to 8×1 (8 fixtures)
  2. Add a GLSL TOP with this shader:
uniform float uTime;
out vec4 fragColor;
void main() {
  float x = vUV.s;
  float brightness = step(mod(x * 8.0 + uTime * 2.0, 1.0), 0.5);
  fragColor = vec4(brightness, 0.2, 0.8, 1.0);
}
  1. Wire absTime.seconds → uTime via a CHOP→parameter expression
  2. Add a DMX Out CHOP in "Convert from TOP" mode
  3. You now have an animated chase on 8 fixtures, running entirely on the GPU
Discussion

How would you add a master dimmer? What's the maximum fixture count before you feel a performance difference between Part A and Part B? Where would you insert per-fixture color control?

05

Building a real patch

30 min

Think in layers first

Before opening TD, answer: how many fixtures, what types? What effects are needed? What's controlled in real time vs pre-programmed? Does it respond to external input?

// diagram — patch architecture layers

DAT-based patch table

Build your fixture database in a Table DAT. When you repatch a fixture, you change one cell — not rewire half your network.

Table DAT — fixture patch
fixture_id | name      | address | universe | channels | type
1          | PAR_L1    | 1       | 0        | 4        | RGBW
2          | PAR_L2    | 5       | 0        | 4        | RGBW
3          | TUBE_01   | 9       | 0        | 24       | RGB_PIXEL_8
4          | MH_01     | 33      | 0        | 16       | MOVING_HEAD

Generate your Art-Net routing from this table with a Script DAT. A Python extension can expose methods like getFixtureChannels('PAR_L1') for other parts of the patch.

Master dimmer

Always have one — a single float (0–1) that multiplies your entire output. Essential for: fade-to-black, emergency blackout, smooth scene transitions.

  • Hard blackout: all channels to 0 instantly — use for cue transitions, safety
  • Fade: lerp to 0 over N seconds — put a Lag CHOP or Filter CHOP on the master value
  • Implement as a Math CHOP at the end of your chain, or as a uniform in your GLSL shader

Cue system basics

A cue = a snapshot of channel states + a transition time. Simple TD implementation:

  • Store cues as rows in a Table DAT (channel values + fade time)
  • On cue trigger, lerp current values to target over the cue's fade time
  • Use a Timer CHOP to drive the lerp progress
  • For locked show timelines: use LTC In CHOP or Art-Net timecode as the trigger — not wall clock
Pro tip

Build a monitoring panel that shows your expected output values. Art-Net is one-way — you can't ask a fixture what value it's receiving. Your monitoring panel is your only window into what's actually happening downstream.

06

Audio-reactive & generative

25 min

The key distinction

// diagram — audio-reactive vs generative

"Most things people call generative are actually reactive. If it's just noise plugged into noise — that's a patch, not a system."

Getting audio into TD

  • Audio Device In CHOP — real-time mic or line input. Use for live performance.
  • Audio File In CHOP — pre-recorded file. Use for fixed-timeline shows or testing.
  • Ableton Link CHOP — beat-locked sync with Ableton/Bitwig. Gives you BPM, beat phase, bar phase. More reliable than beat detection for live performance.

Mapping audio to lights

  • Simple brightness: low-frequency band → master dimmer. Lights pulse with the beat. Add a Lag CHOP for attack/release shaping.
  • Color mapping: low freq → R, mid → G, high → B. Color shifts with spectral content.
  • Beat-locked chase: Beat CHOP gives 0–1 ramp per beat. Use to index into fixture sequence.

Building actual generative systems

Two-layer approach: an autonomous system that's always running and interesting on its own, plus audio as a modulation source — not a direct driver.

Example: a reaction-diffusion system where the feed rate is influenced by kick drum energy. The system keeps going; the audio makes it react visibly without reducing it to a simple flash.

Exercise 03 Audio-reactive chase 20 min
  1. Create an Audio Device In CHOP — set to your mic or system audio
  2. Add an Audio Spectrum CHOP — 8 bands, smoothing 0.3. Observe the 8 output channels responding to audio.
  3. Add a Select CHOP to grab just channel 0 (low frequency / kick)
  4. Add a Lag CHOP — attack 0.01, release 0.3 (fast on, slow off). This is your kick detector.
  5. Create a Pattern CHOP — Ramp type, 8 samples (for 8 fixtures)
  6. Add a Math CHOP to offset the ramp position using a counter driven by your kick signal
  7. Multiply the ramp output by your kick detector value — brightness pulses with the beat, position advances on each kick
  8. Wire to Art-Net Out CHOP — watch your 8 fixtures chase in time with audio
Tuning discussion

What does changing the Lag release time do to the feel? What happens if you use a high-frequency band instead of low? How would you add color (not just brightness) to the chase? How would you make the speed BPM-locked?

07

Production & real-world considerations

15 min

Network architecture on shows

  • Dedicated lighting network — never share with internet, video, or production network
  • Managed switches only — unmanaged switches drop Art-Net packets under load
  • Static IPs for everything — DHCP address changes on reboot = catastrophic during a show
  • IGMP snooping — if using sACN and fixtures disappear, check this on your switch config
  • Fiber for long runs — copper ethernet max 100m; fiber eliminates ground loops

Redundancy

  • On important shows: two TD machines, one hot standby
  • Always have a backup scene that can be triggered instantly — a simple, stable state
  • Art-Net merge: some nodes accept two sources and merge — useful for redundant systems
  • Enable spanning tree protocol on switches to prevent network loops

Working with lighting designers

  • LDs think in cues, scenes, and groups — learn this vocabulary
  • If they're on a console (MA3, Eos) and TD is a node in their system: you'll receive OSC or Art-Net triggers
  • If TD is primary control: export your patch data in a format they can import (CSV, MVR)
  • Be explicit about what TD controls vs what the console controls — shared universes cause conflicts

Appendix C — GLSL snippets

RGB → RGBW conversion
vec3 rgb = texture(sTD2DInputs[0], vUV.st).rgb;
float w     = min(rgb.r, min(rgb.g, rgb.b));
vec3 rgbOut = rgb - vec3(w);
fragColor   = vec4(rgbOut, w);
// R, G, B channels reduced + W channel added
Animated chase pattern
uniform float uTime;
uniform float uSpeed;
out vec4 fragColor;
void main() {
  float index      = vUV.s;
  float phase      = mod(index - uTime * uSpeed, 1.0);
  float brightness = step(0.5, phase);
  fragColor        = vec4(brightness, 0.2, 0.8, 1.0);
}
16-bit DMX output (MSB / LSB split for pan/tilt)
// value: 0.0–1.0 input
// Outputs two channels: MSB (coarse) and LSB (fine)
float value16 = clamp(value, 0.0, 1.0) * 65535.0;
float msb     = floor(value16 / 256.0) / 255.0;
float lsb     = mod(value16, 256.0)    / 255.0;
Audio-reactive brightness with Lag
// In a CHOP network:
// Audio Spectrum → Select (ch0) → Lag (att:0.01 rel:0.3)
//   → Math (multiply your color TOP) → DMX Out

// Or in GLSL as a uniform:
uniform float uKick;   // fed from Lag CHOP via parameter
out vec4 fragColor;
void main() {
  vec3 color    = texture(sTD2DInputs[0], vUV.st).rgb;
  fragColor     = vec4(color * uKick, 1.0);
}
